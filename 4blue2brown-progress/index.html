Why this route

Motivation

The goal is not to copy a channel identity. The goal is to learn from the Manim production model: explanations should be precise, editable, rerenderable, and easy to inspect. Raw video generation can look plausible, but it is still hard to verify, repair, and ground in formulas or source code.

A paper-with-code explainer needs four synchronized channels: the paper claim, the formula, the implementation, and a toy example or comparison. If one channel is missing, the animation may look polished while teaching the wrong thing.

Current thesis

Keep code as the controllable substrate, then use agents around it: planning before code, render logs for repair, frame probes for visual critique, and learner questions for evaluation.

  • Plan scenes before writing Manim.
  • Treat the renderer as a hard gate, not a final quality score.
  • Ask VLMs to review sampled frames, not imagined outputs.
  • Bind claims, formulas, code paths, toy data, and visual beats in one graph.

Slides are not a replacement for Manim, but they are a useful adjacent substrate. They are editable, easy to skim, easier to evaluate as pages, and closer to how researchers already explain a paper. What slides can add

  • Deck outline is a natural planning intermediate before Manim scenes.
  • Editable pages expose layout decisions that video clips hide.
  • Speaker notes can become narration drafts for later video generation.
  • Slide-level citations make paper/code grounding easier to audit.

What still needs our pipeline

  • Code grounding is weak in most slide systems by default.
  • Formula-to-implementation mapping needs a PaperCodeGraph-like artifact.
  • Slides must distinguish source figures from generated decorations.
  • Evaluation must measure teaching value, not only design polish.

Separate model comparison

Model Comparison Plan

Model comparison should not be mixed with framework comparison. First keep one framework fixed and swap the model family. Then keep the topic fixed and swap the framework style. Code2Video is the most stable rendered path today, so it is the first practical harness.

Experiment A: same framework, different model

Run Code2Video with one GPT-family, one Claude-family, and one Gemini-family model on the same three specs: Transformers, DPO, and RoPE.

  • Keep prompts, max sections, quality, and no-LaTeX constraints fixed.
  • Log generation wall time, render wall time, retries, tokens, and dollar cost.
  • Review every output with the same contact-sheet rubric and a human pass.

Experiment B: same topic, different framework

After Experiment A, compare Paper2Manim, Code2Video, TEA-style, LLM2Manim, and ManimTrainer-style outputs for each of the five clip topics. This isolates framework bias from model bias.

  • Expect Paper2Manim to produce better scene decomposition.
  • Expect Code2Video to produce better runnable code sooner.
  • Expect TEA, LLM2Manim, and ManimTrainer styles to expose planning, pedagogy, and training-loop bias.

Quality gates

Evaluation Stack

Render success is only the first gate. A useful clip also needs readable geometry, correct formulas, grounded code references, and evidence that a learner understands more after seeing it.

Implemented now

  • ffprobe metadata: duration, size, resolution, FPS.
  • Frame sampler that avoids fade-transition false negatives.
  • Contact sheets for manual and multimodal review.
  • WInE/Gemini VLM review over rendered keyframes.
  • Structured notes for "what works" and "what fails."

Needed next

  • Scene IR validator for symbols, objects, and timing.
  • Formula-code grounding checks for FeynRL/P3O.
  • Layout checks for overlap, tiny text, and offscreen objects.
  • TeachQuiz or learner questions per clip.
  • Memory bank that stores validated successes and failures separately.

Geometry and aesthetics need measurable signals

The biggest visible gap is not syntax. Models can render Manim while still failing at spatial hierarchy: tiny formulas, unclear alignment, static text, limited motion, and diagrams that do not reveal the mechanism. We need geometry metrics before asking a judge for taste.

  • OCR text size and line density per frame.
  • Bounding-box overlap, offscreen objects, and crowded safe areas.
  • Motion signal: did the key object move, morph, or only appear as text?
  • Alignment score for axes, arrows, formulas, and highlighted objects.

Learner-bench signal

The more interesting metric is not "does a judge like the video?" It is whether a learner can answer paper-specific questions after seeing the generated artifact. Talkie-LM is a useful reference: a model with an old knowledge cutoff creates a cleaner generalization test because it cannot simply rely on modern web contamination.

  • Build paper questions before video generation, then hide the paper.
  • Give a learner model only keyframes, transcript, and code highlights.
  • Measure pre/post improvement, misconception reduction, and citation grounding.
  • Use learning gain as an RL/reward signal, not just VLM preference.

Reference: talkie, a pre-1931 vintage LM

Future direction

Next Milestones

The next phase is to turn a promising progress site into a repeatable experiment harness. FeynRL/P3O is the best first full target because it has formulas, code paths, baselines, and a system-level idea, replay staleness, that should be visible on screen.

  1. M1 Paper/code index

    Cache formulas, claims, code symbols, configs, and examples. Success: answer "where is this formula in code?" for FeynRL.

  2. M2 Scene IR

    YAML plans for FeynRL method scenes: ratios, ESS, fixed clipping, CISPO, P3O, sync vs async.

  3. M3 First grounded Manim clip

    One 10-30s clip explaining ESS tradeoff with formula, toy batch, and code pointer.

  4. M4 Comparison clip

    PPO/GRPO/CISPO/P3O side-by-side under fresh, mildly stale, and stale ratio regimes.

  5. M5 Findings clip

    Turn one paper claim into a visual explanation backed by code, example, and result.

  6. M6 One-command rough video

    `generate` produces clips, review artifacts, narration draft, subtitles, and a reproducible run directory.
